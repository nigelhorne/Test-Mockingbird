   77: =head1 DESCRIPTION
   78: 
   79: Test::Mockingbird provides powerful mocking, spying, and dependency injection capabilities to streamline testing in Perl.
   80: 
   81: =head1 DIAGNOSTICS
   82: 
   83: Test::Mockingbird provides optional, non-intrusive diagnostic routines
   84: that allow inspection of the current mocking state during test execution.
   85: These routines are purely observational. They do not modify any mocking
   86: behaviour, symbol table entries, or internal state.
   87: 
   88: Diagnostics are useful when debugging complex test suites, verifying
   89: mock layering behaviour, or understanding interactions between multiple
   90: mocking primitives such as mock, spy, inject, and the sugar functions.
   91: 
   92: =head2 diagnose_mocks
   93: 
   94: Return a structured hashref describing all currently active mock layers.
   95: Each entry includes the fully qualified method name, the number of active
   96: layers, whether the original method existed, and metadata for each layer
   97: (type and installation location). See the diagnose_mocks method for full
   98: API details.
   99: 
  100: =head2 diagnose_mocks_pretty
  101: 
  102: Return a human-readable, multi-line string describing all active mock
  103: layers. This routine is intended for debugging and inspection during test
  104: development. The output format is stable for human consumption but is not
  105: guaranteed for machine parsing. See the diagnose_mocks_pretty method for
  106: full API details.
  107: 
  108: =head2 Diagnostic Metadata
  109: 
  110: Diagnostic information is recorded automatically whenever a mock layer is
  111: successfully installed. Each layer records:
  112: 
  113:   * type          The category of mock layer (for example: mock, spy,
  114:                   inject, mock_return, mock_exception, mock_sequence,
  115:                   mock_once, mock_scoped)
  116: 
  117:   * installed_at  The file and line number where the layer was created
  118: 
  119: This metadata is maintained in parallel with the internal mock stack and
  120: is automatically cleared when a method is fully restored via unmock or
  121: restore_all.
  122: 
  123: Diagnostics never alter the behaviour of the mocking engine and may be
  124: safely invoked at any point during a test run.
  125:

  232: =head2 mock($package, $method, $replacement)
  233: 
  234: Mocks a method in the specified package.
  235: Supports two forms:
  236: 
  237:     mock('My::Module', 'method', sub { ... });
  238: 
  239: or the shorthand:
  240: 
  241:     mock 'My::Module::method' => sub { ... };

  245: This prevents Perl from emitting C<Prototype mismatch> warnings at call

  253: replacement.

  355: =head2 mock_scoped
  356: 
  357: Creates a scoped mock that is automatically restored when the returned guard
  358: goes out of scope.
  359: 
  360: This behaves like C<mock>, but instead of requiring an explicit call to
  361: C<unmock> or C<restore_all>, all mocked methods are reverted automatically
  362: when the guard object is destroyed.
  363: 
  364: =head3 Single-method forms
  365: 
  366: Shorthand:
  367: 
  368:     # The return value must be assigned to a lexical variable.
  369:     # If discarded, the guard object is destroyed immediately and the mock never takes effect.
  370:     # When $g goes out of scope, all mocked methods are restored automatically.
  371: 
  372:     my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  373: 
  374: Longhand:
  375: 
  376:     my $g = mock_scoped('My::Module', 'method', sub { ... });
  377: 
  378: =head3 Multi-method forms
  379: 
  380: Mock several methods on one package with a single guard:
  381: 
  382:     my $g = mock_scoped('My::Module',
  383:         fetch  => sub { 'mocked_fetch'  },
  384:         save   => sub { 'mocked_save'   },
  385:         delete => sub { 'mocked_delete' },
  386:     );
  387: 
  388: Mock methods across different packages in one call (shorthand pairs):
  389: 
  390:     my $g = mock_scoped(
  391:         'My::Module::fetch'  => sub { 'mocked_fetch'  },
  392:         'Other::Module::save' => sub { 'mocked_save'  },
  393:     );
  394: 
  395: In both multi-method forms, every mocked method is restored when C<$g>
  396: goes out of scope or is explicitly undefed.
  397: 
  398: =head3 Scoped lifecycle
  399: 
  400:     {
  401:         my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  402:         My::Module::method();   # returns 'mocked'
  403:     }
  404: 
  405:     My::Module::method();       # original behaviour restored
  406: 
  407: =head3 Interaction with spy
  408: 
  409: A C<spy> is not automatically restored when a C<mock_scoped> guard
  410: goes out of scope. C<mock_scoped> only manages the specific mock
  411: layer it installs. If you install a spy inside a scoped block, you
  412: must restore it explicitly:
  413: 
  414:     {
  415:         my $g   = mock_scoped 'My::Module::method' => sub { 1 };
  416:         my $spy = spy 'My::Module::method';
  417: 
  418:         My::Module->method('arg');
  419:     }
  420:     # $g is destroyed here -- the mock_scoped layer is restored
  421:     # but the spy layer is still active
  422: 
  423:     restore_all();    # needed to fully restore method
  424: 
  425: The safe pattern when combining C<mock_scoped> and C<spy> is to
  426: call C<restore_all> at the end of the block, or to avoid combining
  427: them and use C<mock> with an explicit C<restore_all> instead:
  428: 
  429:     spy 'My::Module::method';
  430:     My::Module->method('arg');
  431:     my @calls = $spy->();
  432:     restore_all();
  433: 
  434: =head3 Notes
  435: 
  436: If you need both a modified implementation and call recording in
  437: the same test, install the spy first and then the mock. The spy
  438: will still capture calls even when the implementation is replaced
  439: by the mock layer above it, because the spy wraps the layer below
  440: it at installation time, not the current top of the stack. To avoid
  441: confusion, prefer explicit C<restore_all> over C<mock_scoped> when
  442: combining with spies.
  443:

  525: =head2 spy($package, $method)
  526: 
  527: Wraps a method so that all calls and arguments are recorded.
  528: Supports two forms:
  529: 
  530:     spy('My::Module', 'method');
  531: 
  532: or the shorthand:
  533: 
  534:     spy 'My::Module::method';
  535: 
  536: Returns a coderef which, when invoked, returns the list of captured calls.
  537: The original method is preserved and still executed.
  538: 
  539: =head3 Call record format
  540: 
  541: Each captured call is an arrayref with the following structure:
  542: 
  543:     [ $method_name, $invocant, @arguments ]
  544: 
  545: where:
  546: 
  547: =over 4
  548: 
  549: =item * C<$method_name> - the fully qualified method name as a string
  550: (e.g. C<'My::Module::method'>)
  551: 
  552: =item * C<$invocant> - the first argument to the call, typically C<$self>
  553: for method calls or the first positional argument for function calls
  554: 
  555: =item * C<@arguments> - the remaining arguments passed to the method,
  556: in the order they were supplied.
  557: For named-parameter calls,
  558: these will be alternating key/value pairs suitable for assignment to a hash:
  559: C<my %args = @{$call}[2..$#{$call}]>
  560: 
  561: =back
  562: 
  563: =head3 Example
  564: 
  565:     spy 'My::Module::process';
  566:     My::Module->process(name => 'foo', value => 42);
  567: 
  568:     my @calls = $spy->();
  569:     my $call  = $calls[0];
  570: 
  571:     # $call->[0] eq 'My::Module::process'
  572:     # $call->[1] is the My::Module object
  573:     # @{$call}[2..$#{$call}] gives (name => 'foo', value => 42)
  574: 
  575:     my %args = @{$call}[2..$#{$call}];
  576:     is($args{name},  'foo', 'name arg captured');
  577:     is($args{value}, 42,    'value arg captured');
  578: 
  579: =head3 Limitations
  580: 
  581: C<spy> installs its wrapper coderef directly into the glob without going
  582: through C<mock>, so the prototype-preservation logic in C<mock> does not apply.
  583: If the target function carries a Perl prototype (for example, a
  584: C<()> no-args prototype), installing a spy will emit a
  585: C<Prototype mismatch> warning.
  586: 
  587: If you need warning-free wrapping of a prototyped function, install the
  588: spy on a non-prototyped alias, or use C<mock> with a wrapper that records
  589: calls and delegates to the original:
  590: 
  591:     my @calls;
  592:     mock 'My::Module::detect' => sub {
  593:         push @calls, [@_];
  594:         return My::Module::_real_detect(@_);   # delegate manually
  595:     };
  596: 
  597: This limitation will be addressed in a future release.
  598:
