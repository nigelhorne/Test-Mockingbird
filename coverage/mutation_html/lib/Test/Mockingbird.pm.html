   66: =head1 DESCRIPTION
   67: 
   68: Test::Mockingbird provides powerful mocking, spying, and dependency injection capabilities to streamline testing in Perl.
   69: 
   70: =head1 DIAGNOSTICS
   71: 
   72: Test::Mockingbird provides optional, non-intrusive diagnostic routines
   73: that allow inspection of the current mocking state during test execution.
   74: These routines are purely observational. They do not modify any mocking
   75: behaviour, symbol table entries, or internal state.
   76: 
   77: Diagnostics are useful when debugging complex test suites, verifying
   78: mock layering behaviour, or understanding interactions between multiple
   79: mocking primitives such as mock, spy, inject, and the sugar functions.
   80: 
   81: =head2 diagnose_mocks
   82: 
   83: Return a structured hashref describing all currently active mock layers.
   84: Each entry includes the fully qualified method name, the number of active
   85: layers, whether the original method existed, and metadata for each layer
   86: (type and installation location). See the diagnose_mocks method for full
   87: API details.
   88: 
   89: =head2 diagnose_mocks_pretty
   90: 
   91: Return a human-readable, multi-line string describing all active mock
   92: layers. This routine is intended for debugging and inspection during test
   93: development. The output format is stable for human consumption but is not
   94: guaranteed for machine parsing. See the diagnose_mocks_pretty method for
   95: full API details.
   96: 
   97: =head2 Diagnostic Metadata
   98: 
   99: Diagnostic information is recorded automatically whenever a mock layer is
  100: successfully installed. Each layer records:
  101: 
  102:   * type          The category of mock layer (for example: mock, spy,
  103:                   inject, mock_return, mock_exception, mock_sequence,
  104:                   mock_once, mock_scoped)
  105: 
  106:   * installed_at  The file and line number where the layer was created
  107: 
  108: This metadata is maintained in parallel with the internal mock stack and
  109: is automatically cleared when a method is fully restored via unmock or
  110: restore_all.
  111: 
  112: Diagnostics never alter the behaviour of the mocking engine and may be
  113: safely invoked at any point during a test run.
  114:

  222: 
  223: Mocks a method in the specified package.
  224: Supports two forms:
  225: 
  226:     mock('My::Module', 'method', sub { ... });

  230:     mock 'My::Module::method' => sub { ... };

  344: =head2 mock_scoped
  345: 
  346: Creates a scoped mock that is automatically restored when the returned guard
  347: goes out of scope.
  348: 
  349: This behaves like C<mock>, but instead of requiring an explicit call to
  350: C<unmock> or C<restore_all>, all mocked methods are reverted automatically
  351: when the guard object is destroyed.
  352: 
  353: =head3 Single-method forms
  354: 
  355: Shorthand:
  356: 
  357:     # The return value must be assigned to a lexical variable.
  358:     # If discarded, the guard object is destroyed immediately and the mock never takes effect.
  359:     # When $g goes out of scope, all mocked methods are restored automatically.
  360: 
  361:     my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  362: 
  363: Longhand:
  364: 
  365:     my $g = mock_scoped('My::Module', 'method', sub { ... });
  366: 
  367: =head3 Multi-method forms
  368: 
  369: Mock several methods on one package with a single guard:
  370: 
  371:     my $g = mock_scoped('My::Module',
  372:         fetch  => sub { 'mocked_fetch'  },
  373:         save   => sub { 'mocked_save'   },
  374:         delete => sub { 'mocked_delete' },
  375:     );
  376: 
  377: Mock methods across different packages in one call (shorthand pairs):
  378: 
  379:     my $g = mock_scoped(
  380:         'My::Module::fetch'  => sub { 'mocked_fetch'  },
  381:         'Other::Module::save' => sub { 'mocked_save'  },
  382:     );
  383: 
  384: In both multi-method forms, every mocked method is restored when C<$g>
  385: goes out of scope or is explicitly undefed.
  386: 
  387: =head3 Scoped lifecycle
  388: 
  389:     {
  390:         my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  391:         My::Module::method();   # returns 'mocked'
  392:     }
  393: 
  394:     My::Module::method();       # original behaviour restored
  395: 
  396: =head3 Interaction with spy
  397: 
  398: A C<spy> is not automatically restored when a C<mock_scoped> guard
  399: goes out of scope. C<mock_scoped> only manages the specific mock
  400: layer it installs. If you install a spy inside a scoped block, you
  401: must restore it explicitly:
  402: 
  403:     {
  404:         my $g   = mock_scoped 'My::Module::method' => sub { 1 };
  405:         my $spy = spy 'My::Module::method';
  406: 
  407:         My::Module->method('arg');
  408:     }
  409:     # $g is destroyed here -- the mock_scoped layer is restored
  410:     # but the spy layer is still active
  411: 
  412:     restore_all();    # needed to fully restore method
  413: 
  414: The safe pattern when combining C<mock_scoped> and C<spy> is to
  415: call C<restore_all> at the end of the block, or to avoid combining
  416: them and use C<mock> with an explicit C<restore_all> instead:
  417: 
  418:     spy 'My::Module::method';
  419:     My::Module->method('arg');
  420:     my @calls = $spy->();
  421:     restore_all();
  422: 
  423: =head3 Notes
  424: 
  425: If you need both a modified implementation and call recording in
  426: the same test, install the spy first and then the mock. The spy
  427: will still capture calls even when the implementation is replaced
  428: by the mock layer above it, because the spy wraps the layer below
  429: it at installation time, not the current top of the stack. To avoid
  430: confusion, prefer explicit C<restore_all> over C<mock_scoped> when
  431: combining with spies.

  514: =head2 spy($package, $method)
  515: 
  516: Wraps a method so that all calls and arguments are recorded.
  517: Supports two forms:
  518: 
  519:     spy('My::Module', 'method');
  520: 
  521: or the shorthand:
  522: 
  523:     spy 'My::Module::method';
  524: 
  525: Returns a coderef which, when invoked, returns the list of captured calls.
  526: The original method is preserved and still executed.
  527: 
  528: =head3 Call record format
  529: 
  530: Each captured call is an arrayref with the following structure:
  531: 
  532:     [ $method_name, $invocant, @arguments ]
  533: 
  534: where:
  535: 
  536: =over 4
  537: 
  538: =item * C<$method_name> - the fully qualified method name as a string
  539: (e.g. C<'My::Module::method'>)
  540: 
  541: =item * C<$invocant> - the first argument to the call, typically C<$self>
