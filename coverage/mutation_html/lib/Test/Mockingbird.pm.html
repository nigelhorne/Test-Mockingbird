   66: =head1 DESCRIPTION
   67: 
   68: Test::Mockingbird provides powerful mocking, spying, and dependency injection capabilities to streamline testing in Perl.
   69: 
   70: =head1 DIAGNOSTICS
   71: 
   72: Test::Mockingbird provides optional, non-intrusive diagnostic routines
   73: that allow inspection of the current mocking state during test execution.
   74: These routines are purely observational. They do not modify any mocking
   75: behaviour, symbol table entries, or internal state.
   76: 
   77: Diagnostics are useful when debugging complex test suites, verifying
   78: mock layering behaviour, or understanding interactions between multiple
   79: mocking primitives such as mock, spy, inject, and the sugar functions.
   80: 
   81: =head2 diagnose_mocks
   82: 
   83: Return a structured hashref describing all currently active mock layers.
   84: Each entry includes the fully qualified method name, the number of active
   85: layers, whether the original method existed, and metadata for each layer
   86: (type and installation location). See the diagnose_mocks method for full
   87: API details.
   88: 
   89: =head2 diagnose_mocks_pretty
   90: 
   91: Return a human-readable, multi-line string describing all active mock
   92: layers. This routine is intended for debugging and inspection during test
   93: development. The output format is stable for human consumption but is not
   94: guaranteed for machine parsing. See the diagnose_mocks_pretty method for
   95: full API details.
   96: 
   97: =head2 Diagnostic Metadata
   98: 
   99: Diagnostic information is recorded automatically whenever a mock layer is
  100: successfully installed. Each layer records:
  101: 
  102:   * type          The category of mock layer (for example: mock, spy,
  103:                   inject, mock_return, mock_exception, mock_sequence,
  104:                   mock_once, mock_scoped)
  105: 
  106:   * installed_at  The file and line number where the layer was created
  107: 
  108: This metadata is maintained in parallel with the internal mock stack and
  109: is automatically cleared when a method is fully restored via unmock or
  110: restore_all.
  111: 
  112: Diagnostics never alter the behaviour of the mocking engine and may be
  113: safely invoked at any point during a test run.
  114:

  222: 
  223: Mocks a method in the specified package.
  224: Supports two forms:
  225: 
  226:     mock('My::Module', 'method', sub { ... });

  230:     mock 'My::Module::method' => sub { ... };

  344: =head2 mock_scoped
  345: 
  346: Creates a scoped mock that is automatically restored when the returned guard
  347: goes out of scope.
  348: 
  349: This behaves like C<mock>, but instead of requiring an explicit call to
  350: C<unmock> or C<restore_all>, all mocked methods are reverted automatically
  351: when the guard object is destroyed.
  352: 
  353: =head3 Single-method forms
  354: 
  355: Shorthand:
  356: 
  357:     my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  358: 
  359: Longhand:
  360: 
  361:     my $g = mock_scoped('My::Module', 'method', sub { ... });
  362: 
  363: =head3 Multi-method forms
  364: 
  365: Mock several methods on one package with a single guard:
  366: 
  367:     my $g = mock_scoped('My::Module',
  368:         fetch  => sub { 'mocked_fetch'  },
  369:         save   => sub { 'mocked_save'   },
  370:         delete => sub { 'mocked_delete' },
  371:     );
  372: 
  373: Mock methods across different packages in one call (shorthand pairs):
  374: 
  375:     my $g = mock_scoped(
  376:         'My::Module::fetch'  => sub { 'mocked_fetch'  },
  377:         'Other::Module::save' => sub { 'mocked_save'  },
  378:     );
  379: 
  380: In both multi-method forms, every mocked method is restored when C<$g>
  381: goes out of scope or is explicitly undefed.
  382: 
  383: =head3 Scoped lifecycle
  384: 
  385:     {
  386:         my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  387:         My::Module::method();   # returns 'mocked'
  388:     }
  389: 
  390:     My::Module::method();       # original behaviour restored
  391: 
  392: =head3 Interaction with spy
  393: 
  394: A C<spy> is not automatically restored when a C<mock_scoped> guard
  395: goes out of scope. C<mock_scoped> only manages the specific mock
  396: layer it installs. If you install a spy inside a scoped block, you
  397: must restore it explicitly:
  398: 
  399:     {
  400:         my $g   = mock_scoped 'My::Module::method' => sub { 1 };
  401:         my $spy = spy 'My::Module::method';
  402: 
  403:         My::Module->method('arg');
  404:     }
  405:     # $g is destroyed here -- the mock_scoped layer is restored
  406:     # but the spy layer is still active
  407: 
  408:     restore_all();    # needed to fully restore method
  409: 
  410: The safe pattern when combining C<mock_scoped> and C<spy> is to
  411: call C<restore_all> at the end of the block, or to avoid combining
  412: them and use C<mock> with an explicit C<restore_all> instead:
  413: 
  414:     spy 'My::Module::method';
  415:     My::Module->method('arg');
  416:     my @calls = $spy->();
  417:     restore_all();
  418: 
  419: =head3 Notes
  420: 
  421: If you need both a modified implementation and call recording in
  422: the same test, install the spy first and then the mock. The spy
  423: will still capture calls even when the implementation is replaced
  424: by the mock layer above it, because the spy wraps the layer below
  425: it at installation time, not the current top of the stack. To avoid
  426: confusion, prefer explicit C<restore_all> over C<mock_scoped> when
  427: combining with spies.

  510: =head2 spy($package, $method)
  511: 
  512: Wraps a method so that all calls and arguments are recorded.
  513: Supports two forms:
  514: 
  515:     spy('My::Module', 'method');
  516: 
  517: or the shorthand:
  518: 
  519:     spy 'My::Module::method';
  520: 
  521: Returns a coderef which, when invoked, returns the list of captured calls.
  522: The original method is preserved and still executed.
  523: 
  524: =head3 Call record format
  525: 
  526: Each captured call is an arrayref with the following structure:
  527: 
  528:     [ $method_name, $invocant, @arguments ]
  529: 
  530: where:
  531: 
  532: =over 4
  533: 
  534: =item * C<$method_name> - the fully qualified method name as a string
  535: (e.g. C<'My::Module::method'>)
  536: 
  537: =item * C<$invocant> - the first argument to the call, typically C<$self>
  538: for method calls or the first positional argument for function calls
  539: 
  540: =item * C<@arguments> - the remaining arguments passed to the method,
  541: in the order they were supplied. For named-parameter calls these will
  542: be alternating key/value pairs suitable for assignment to a hash:
