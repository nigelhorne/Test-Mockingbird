  102: =head1 DIAGNOSTICS
  103: 
  104: Test::Mockingbird provides optional, non-intrusive diagnostic routines
  105: that allow inspection of the current mocking state during test execution.
  106: These routines are purely observational. They do not modify any mocking
  107: behaviour, symbol table entries, or internal state.
  108: 
  109: Diagnostics are useful when debugging complex test suites, verifying
  110: mock layering behaviour, or understanding interactions between multiple
  111: mocking primitives such as mock, spy, inject, and the sugar functions.
  112: 
  113: =head2 diagnose_mocks
  114: 
  115: Return a structured hashref describing all currently active mock layers.
  116: Each entry includes the fully qualified method name, the number of active
  117: layers, whether the original method existed, and metadata for each layer
  118: (type and installation location). See the diagnose_mocks method for full
  119: API details.
  120: 
  121: =head2 diagnose_mocks_pretty
  122: 
  123: Return a human-readable, multi-line string describing all active mock
  124: layers. This routine is intended for debugging and inspection during test
  125: development. The output format is stable for human consumption but is not
  126: guaranteed for machine parsing. See the diagnose_mocks_pretty method for
  127: full API details.
  128: 
  129: =head2 Diagnostic Metadata
  130: 
  131: Diagnostic information is recorded automatically whenever a mock layer is
  132: successfully installed. Each layer records:
  133: 
  134:   * type          The category of mock layer (for example: mock, spy,
  135:                   inject, mock_return, mock_exception, mock_sequence,
  136:                   mock_once, mock_scoped)
  137: 
  138:   * installed_at  The file and line number where the layer was created
  139: 
  140: This metadata is maintained in parallel with the internal mock stack and
  141: is automatically cleared when a method is fully restored via unmock or
  142: restore_all.
  143: 
  144: Diagnostics never alter the behaviour of the mocking engine and may be
  145: safely invoked at any point during a test run.

  376: =head2 mock_scoped
  377: 
  378: Creates a scoped mock that is automatically restored when the returned guard
  379: goes out of scope.
  380: 
  381: This behaves like C<mock>, but instead of requiring an explicit call to
  382: C<unmock> or C<restore_all>, all mocked methods are reverted automatically
  383: when the guard object is destroyed.
  384: 
  385: =head3 Single-method forms
  386: 
  387: Shorthand:
  388: 
  389:     # The return value must be assigned to a lexical variable.
  390:     # If discarded, the guard object is destroyed immediately and the mock never takes effect.
  391:     # When $g goes out of scope, all mocked methods are restored automatically.
  392: 
  393:     my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  394: 
  395: Longhand:
  396: 
  397:     my $g = mock_scoped('My::Module', 'method', sub { ... });
  398:

  400: 
  401: Mock several methods on one package with a single guard:
  402: 
  403:     my $g = mock_scoped('My::Module',
  404:         fetch  => sub { 'mocked_fetch'  },
  405:         save   => sub { 'mocked_save'   },
  406:         delete => sub { 'mocked_delete' },
  407:     );
  408: 
  409: Mock methods across different packages in one call (shorthand pairs):
  410: 
  411:     my $g = mock_scoped(
  412:         'My::Module::fetch'  => sub { 'mocked_fetch'  },
  413:         'Other::Module::save' => sub { 'mocked_save'  },
  414:     );
  415: 
  416: In both multi-method forms, every mocked method is restored when C<$g>
  417: goes out of scope or is explicitly undefed.
  418: 
  419: =head3 Scoped lifecycle
  420: 
  421:     {
  422:         my $g = mock_scoped 'My::Module::method' => sub { 'mocked' };
  423:         My::Module::method();   # returns 'mocked'
  424:     }
  425: 
  426:     My::Module::method();       # original behaviour restored
  427: 
  428: =head3 Interaction with spy
  429: 
  430: A C<spy> is not automatically restored when a C<mock_scoped> guard
  431: goes out of scope. C<mock_scoped> only manages the specific mock
  432: layer it installs. If you install a spy inside a scoped block, you
  433: must restore it explicitly:
  434: 
  435:     {
  436:         my $g   = mock_scoped 'My::Module::method' => sub { 1 };
  437:         my $spy = spy 'My::Module::method';
  438: 
  439:         My::Module->method('arg');
  440:     }
  441:     # $g is destroyed here -- the mock_scoped layer is restored
  442:     # but the spy layer is still active
  443: 
  444:     restore_all();    # needed to fully restore method
  445: 
  446: The safe pattern when combining C<mock_scoped> and C<spy> is to
  447: call C<restore_all> at the end of the block, or to avoid combining
  448: them and use C<mock> with an explicit C<restore_all> instead:
  449: 
  450:     spy 'My::Module::method';
  451:     My::Module->method('arg');
  452:     my @calls = $spy->();
  453:     restore_all();
  454: 
  455: =head3 Notes
  456: 
  457: If you need both a modified implementation and call recording in
  458: the same test, install the spy first and then the mock. The spy
  459: will still capture calls even when the implementation is replaced
  460: by the mock layer above it, because the spy wraps the layer below
  461: it at installation time, not the current top of the stack. To avoid
  462: confusion, prefer explicit C<restore_all> over C<mock_scoped> when
  463: combining with spies.

  546: =head2 spy($package, $method)
  547: 
  548: Wraps a method so that all calls and arguments are recorded.
  549: Supports two forms:
  550: 
  551:     spy('My::Module', 'method');
  552: 
  553: or the shorthand:
  554: 
  555:     spy 'My::Module::method';
  556: 
  557: Returns a coderef which, when invoked, returns the list of captured calls.
  558: The original method is preserved and still executed.
  559: 
  560: =head3 Call record format
  561: 
  562: Each captured call is an arrayref with the following structure:
  563: 
  564:     [ $method_name, $invocant, @arguments ]
  565: 
  566: where:
  567: 
  568: =over 4
  569: 
  570: =item * C<$method_name> - the fully qualified method name as a string
  571: (e.g. C<'My::Module::method'>)
  572: 
  573: =item * C<$invocant> - the first argument to the call, typically C<$self>
  574: for method calls or the first positional argument for function calls
  575: 
  576: =item * C<@arguments> - the remaining arguments passed to the method,
  577: in the order they were supplied.
  578: For named-parameter calls,
  579: these will be alternating key/value pairs suitable for assignment to a hash:
  580: C<my %args = @{$call}[2..$#{$call}]>
  581: 
  582: =back
  583: 
  584: =head3 Example
  585: 
  586:     spy 'My::Module::process';
  587:     My::Module->process(name => 'foo', value => 42);
  588: 
  589:     my @calls = $spy->();
  590:     my $call  = $calls[0];
  591: 
  592:     # $call->[0] eq 'My::Module::process'
  593:     # $call->[1] is the My::Module object
  594:     # @{$call}[2..$#{$call}] gives (name => 'foo', value => 42)
  595: 
  596:     my %args = @{$call}[2..$#{$call}];
  597:     is($args{name},  'foo', 'name arg captured');
  598:     is($args{value}, 42,    'value arg captured');
