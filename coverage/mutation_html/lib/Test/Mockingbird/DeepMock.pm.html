   63: =head1 DESCRIPTION
   64: 
   65: C<Test::Mockingbird::DeepMock> provides a declarative, data-driven way to
   66: describe mocking, spying, injection, and expectations in Perl tests.
   67: 
   68: Instead of scattering C<mock>, C<spy>, and C<restore_all> calls throughout
   69: your test code, DeepMock lets you define a complete mocking plan in a single
   70: hashref, then executes your test code under that plan.
   71: 
   72: This produces tests that are:
   73: 
   74: =over 4
   75: 
   76: =item * easier to read
   77: 
   78: =item * easier to maintain
   79: 
   80: =item * easier to extend
   81: 
   82: =item * easier to reason about
   83: 
   84: =back
   85: 
   86: DeepMock is built on top of L<Test::Mockingbird>, adding structure,
   87: expectations, and a clean DSL.
   88: 
   89: =head1 WHY DEEP MOCK?
   90: 
   91: Traditional mocking in Perl tends to be:
   92: 
   93: =over 4
   94: 
   95: =item * imperative
   96: 
   97: =item * scattered across the test body
   98: 
   99: =item * difficult to audit
  100: 
  101: =item * easy to forget to restore
  102: 
  103: =back
  104: 
  105: DeepMock solves these problems by letting you declare everything up front:
  106: 
  107:     deep_mock(
  108:         {
  109:             mocks        => [...],
  110:             expectations => [...],
  111:         },
  112:         sub { ... }
  113:     );
  114: 
  115: This gives you:
  116: 
  117: =over 4
  118: 
  119: =item * a single place to see all mocks and spies
  120: 
  121: =item * automatic restore of all mocks
  122: 
  123: =item * structured expectations
  124: 
  125: =item * reusable patterns
  126: 
  127: =item * a clean separation between setup and test logic
  128: 
  129: =back
  130: 
  131: =head1 PLAN STRUCTURE
  132: 
  133: A DeepMock plan is a hashref with the following keys:
  134: 
  135: =head2 C<mocks>
  136: 
  137: An arrayref of mock specifications. Each entry is a hashref:
  138: 
  139:     {
  140:         target => 'Package::method',   # required
  141:         type   => 'mock' | 'spy' | 'inject',
  142:         with   => sub { ... },         # for mock/inject
  143:         tag    => 'identifier',        # for spies or scoped mocks
  144:         scoped => 1,                   # optional
  145:     }
  146: 
  147: =head3 Types
  148: 
  149: =over 4
  150: 
  151: =item C<mock>
  152: 
  153: Replaces the target method with the provided coderef.
  154: 
  155: =item C<spy>
  156: 
  157: Wraps the method and records all calls. Must have a C<tag>.
  158: 
  159: =item C<inject>
  160: 
  161: Injects a value or behavior into the target (delegates to C<Test::Mockingbird::inject>).
  162: 
  163: =back
  164: 
  165: =head2 C<expectations>
  166: 
  167: An arrayref of expectation specifications. Each entry is a hashref:
  168: 
  169:     {
  170:         tag   => 'double_spy',   # required
  171:         calls => 2,              # optional
  172:         args_like => [           # optional
  173:             [ qr/foo/, qr/bar/ ],
  174:         ],
  175:     }
  176: 
  177: =head3 Expectation fields
  178: 
  179: =over 4
  180: 
  181: =item C<tag>
  182: 
  183: Identifies which spy this expectation applies to.
  184: 
  185: =item C<calls>
  186: 
  187: Expected number of calls.
  188: 
  189: =item C<args_eq>
  190: 
  191: Arrayref of arrayrefs. Each inner array lists exact argument values expected
  192: for a specific call. Values are compared with C<Test::More::is>.
  193: 
  194: =item C<args_deeply>
  195: 
  196: Arrayref of arrayrefs. Each inner array lists deep structures to compare
  197: against the arguments for a specific call. Uses C<Test::Deep::cmp_deeply>.
  198: 
  199: =item C<args_like>
  200: 
  201: Arrayref of arrayrefs of regexes. Each inner array describes expected
  202: arguments for a specific call.
  203: 
  204: =item C<never>
  205: 
  206: Asserts that the spy was never called.
  207: Mutually exclusive with C<calls>.
  208: 
  209: =back
  210: 
  211: =head2 C<globals>
  212: 
  213: Optional hashref controlling global behavior:
  214: 
  215:     globals => {
  216:         restore_on_scope_exit => 1,   # default
  217:     }
  218: 
  219: =head2 C<time>
  220: 
  221: Optional hashref describing a time-travel plan to apply while the
  222: C<deep_mock> block is running. This integrates with
  223: L<Test::Mockingbird::TimeTravel> and allows declarative control of frozen
  224: time, time jumps, and temporal overrides.
  225: 
  226: If provided, the time plan is applied:
  227: 
  228: =over 4
  229: 
  230: =item 1. before any mocks are installed
  231: 
  232: =item 2. before the test code block is executed
  233: 
  234: =item 3. automatically restored after the block completes
  235: 
  236: =back
  237: 
  238: A time plan may include any of the following keys:
  239: 
  240:     time => {
  241:         freeze  => '2025-01-01T00:00:00Z',
  242:         travel  => '2025-01-02T12:00:00Z',
  243:         advance => [ 2 => 'minutes' ],
  244:         rewind  => [ 1 => 'hour'    ],
  245:     }
  246: 
  247: =head3 C<freeze>
  248: 
  249: Freezes time at the given timestamp. Accepts any format supported by
  250: C<Test::Mockingbird::TimeTravel>, including:
  251: 
  252: =over 4
  253: 
  254: =item * C<YYYY-MM-DD>
  255: 
  256: =item * C<YYYY-MM-DD HH:MM:SS>
  257: 
  258: =item * C<YYYY-MM-DDTHH:MM:SSZ>
  259: 
  260: =item * raw epoch seconds
  261: 
  262: =back
  263: 
  264: =head3 C<travel>
  265: 
  266: Moves the frozen clock to a new timestamp without unfreezing time.
  267: 
  268: =head3 C<advance>
  269: 
  270: Advances the frozen clock by a duration. Must be an arrayref:
  271: 
  272:     advance => [ $amount => $unit ]
  273: 
  274: Units may be C<seconds>, C<minutes>, C<hours>, or C<days>.
  275: 
  276: =head3 C<rewind>
  277: 
  278: Rewinds the frozen clock by a duration. Same format as C<advance>.
  279: 
  280: =head3 Example
  281: 
  282:     deep_mock(

  284:             time => {
  285:                 freeze  => '2025-01-01T00:00:00Z',
  286:                 advance => [ 2 => 'minutes' ],
  287:             },

  289:                 {

  457: =head1 TROUBLESHOOTING
  458: 
  459: =head2 "Not enough arguments for deep_mock"
  460: 
  461: You are using the BLOCK prototype form:
  462: 
  463:     deep_mock {
  464:         ...
  465:     }, sub { ... };
  466: 
  467: This only works if C<deep_mock> has a C<(&$)> prototype AND the first
  468: argument is a real block, not a hashref.
  469: 
  470: DeepMock uses C<($$)> to avoid Perl's block-vs-hashref ambiguity.
  471: 
  472: Use parentheses instead:
  473: 
  474:     deep_mock(
  475:         { ... },
  476:         sub { ... }
  477:     );
  478: 
  479: =head2 "Type of arg 1 must be block or sub {}"
  480: 
  481: You are still using the BLOCK prototype form. Switch to parentheses.
  482: 
  483: =head2 "Use of uninitialized value in multiplication"
  484: 
  485: Your spied method is being called with no arguments during spy installation.
  486: Make your method robust:
  487: 
  488:     sub double { ($_[1] // 0) * 2 }
  489: 
  490: =head2 My mocks aren't restored
  491: 
  492: Ensure you didn't disable automatic restore:
  493: 
  494:     globals => { restore_on_scope_exit => 0 }
  495: 
  496: =head2 Nested deep_mock scopes are not supported
  497: 
  498: DeepMock installs mocks using L<Test::Mockingbird>, which provides only
  499: global restore semantics via C<restore_all>. Because Test::Mockingbird
  500: does not expose a per-method restore API, DeepMock cannot safely restore
  501: only the mocks installed in an inner scope.
  502: 
  503: As a result, nested calls like:
  504: 
  505:     deep_mock { ... } sub {
  506:         deep_mock { ... } sub {
  507:             ...
  508:         };
  509:     };
  510: 
  511: will cause the inner restore to remove the outer mocks as well.
  512: 
  513: DeepMock therefore does not support nested mocking scopes.
  514:
